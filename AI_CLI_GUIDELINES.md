# 【核心開發規範】禁止使用 AI CLI 執行持續監聽或阻塞型進程指令

本文件旨在規範並解決團隊在使用 AI 自動化編輯器與開發代理工具（例如 Cursor Agent、Aider、Gemini CLI 等）時，頻繁發生的**終端機背景當機與卡死**現象。

---

## 🚨 為什麼 AI CLI 執行後端伺服器時會「完全卡死」？

當你透過 AI 對話視窗下達如 `node server.js`、`npm run dev` 等進程指令時，背後會發生以下衝突：

1. **AI 代理的生命週期（Wait-For-Exit 機制）**：
   AI CLI 工具的運作邏輯是：「在背景衍生（Spawn）一個子進程 ➔ 執行指令 ➔ **等待進程結束並返回 Exit Code (0 或 1)** ➔ 抓取終端機輸出文字 ➔ 生成解答並回報給使用者」。
   
2. **網路伺服器的生命週期（Infinite Listening Loop 機制）**：
   像 Express 這種 Node.js 後端伺服器，一旦成功啟動，就會進入**永久循環監聽狀態（TCP Listening Loop）**，直到手動按下 `Ctrl + C` 或系統關機為止，它**永遠不會主動退出並返回 Exit Code**。

3. **死鎖（Deadlock）後果**：
   AI 代理在背景傻傻地無限期等待伺服器關閉，而你則在畫面上看到 AI 進度條無限旋轉。此時 AI CLI 將無法處理下一個代碼修改請求，也無法更新上下文，最終只能強制執行作業系統層級的 `taskkill /F /IM node.exe` 或關閉整個整合終端機。

---

## 🚫 嚴格禁用的指令清單（請勿讓 AI 代理代勞）

任何會**霸佔終端機畫面、持續更新、或進入持續監聽狀態**的指令，皆不可在 AI 對話區中直接交付執行：

```bash
# 後端伺服器啟動指令
node server.js
nodemon server.js
node --watch server.js

# 前後端開發伺服器啟動指令
npm run dev
npm start
yarn dev
vite

# 持續更新或監控型指令
npm run build -- --watch
docker compose up (未帶 -d 參數時)
```

## 💡 唯一正確的黃金工作流（Golden Workflow）

請將**「代碼編寫」**與**「程式執行」**的控制權明確切割：

```
┌──────────────────────────────────────┐
│        AI CLI / 對話代理工具         │ ➔ 只負責：寫代碼、改設定、安裝依賴套件
└──────────────────────────────────────┘
                   │
                   ▼ (修改完畢，確認亮綠燈)
┌──────────────────────────────────────┐
│     使用者手動開啟常規終端機面板     │ ➔ 實際動手：cd backend && node server.js
└──────────────────────────────────────┘
```

1. **讓 AI 專注於靜態任務**：在對話框中只要求 AI 進行代碼重構、建立資料夾、撰寫 Prompt、修改設定檔，或是下載與更新套件。
2. **使用者手動控管進程**：
   * 在 VS Code 中按下 \`Ctrl + `\` 開啟系統自帶的常規終端機（Regular Terminal）。
   * 開啟 **雙分頁終端機**：
     * **分頁一**：`cd backend` ➔ `node server.js`（專心監聽 Port 5000）。
     * **分頁二**：`cd ..` 或在根目錄 ➔ `npm run dev` / `npm start`（專心跑前端畫面）。
